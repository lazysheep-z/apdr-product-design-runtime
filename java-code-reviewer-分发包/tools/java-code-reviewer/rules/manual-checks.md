# 需人工 / CI 流程检查的规范条目

以下条目来自《后端日常开发检查评估准则》，**无法通过静态扫描 Java 源码自动判定**，需在 Code Review、Gitee MR 或管理流程中人工确认。

## 一、核心红线（-2 分）

| 编号 | 标题 | 检查方式 |
|------|------|----------|
| 一-3 | 业务域物理/逻辑隔离失效 | Code Review：C 端/B 端/公共组件引用关系 |
| 一-4 | 接口请求嵌套调用超过 3 层 | 架构审查 + 链路追踪 |
| 一-9 | 主分支合入违规 | Gitee 分支保护 + MR 必须 Review |
| 一-11 | 服务阻塞（同步慢操作） | 性能测试 + Code Review |

## 二、优化建议（-1 分）

| 编号 | 标题 | 检查方式 |
|------|------|----------|
| 二-1~3 | 数据模型分层/路径/强类型 | Code Review |
| 二-5~12 | 多环境/事务/熔断/冗余等 | Code Review + 架构审查 |
| 二-14~15 | 代码冗余/重复 | SonarQube 或人工 |
| 二-17~18 | Git 提交规范/颗粒度 | commit-msg hook + MR 大小 |
| 二-19 | 关键位置无注释 | Code Review |
| 二-26 | 未使用 Optional 处理空值 | Code Review（Optional 非强制语法） |

## 三、一般合规项（0 分）

| 编号 | 标题 | 检查方式 |
|------|------|----------|
| 三-1~14, 21~29, 31, 35, 38, 41~42 | 风格/测试/目录等 | IDE 格式化 + 测试覆盖率 CI |

## 四~七、非代码条目

周报管理、提测流程、日常交流、加分项 — **不在 java-review 工具范围内**，由管理者在绩效系统中记录。

## 建议配套

1. **Gitee MR 模板**：要求填写自测范围、影响模块（对应 五-2）
2. **commit-msg hook**：校验 `feat(module): 描述` 格式（对应 二-17）
3. **分支保护**：main/master 禁止直接 push（对应 一-9）
